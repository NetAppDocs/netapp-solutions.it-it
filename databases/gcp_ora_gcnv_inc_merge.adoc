:hardbreaks:
:allow-uri-read: 
:nofooter: 
:icons: font

Allen Cao, Niyaz Mohamed, NetApp

[role="lead"]
La soluzione fornisce una panoramica e dettagli per il rapido ripristino di Oracle VLDB distribuito su Google Compute Engine VM con montaggio NFS sullo storage pool Google Cloud NetApp Volumes per organizzare una copia del database in standby che viene unita in modo incrementale e costante tramite RMAN.



== Scopo

Il ripristino di un database molto grande (VLDB) in Oracle utilizzando lo strumento di backup di Oracle Recovery Manager (RMAN) può essere un'attività molto complessa. Il processo di ripristino del database dai supporti di backup in caso di errore può richiedere molto tempo, ritardando il ripristino del database e potenzialmente compromettendo significativamente il contratto SLA (Service Level Agreement). La situazione sta peggiorando ulteriormente per Oracle VLDB distribuiti nel cloud, poiché il backup RMAN viene generalmente trasferito in streaming su storage a oggetti per motivi economici. Di conseguenza, non è possibile ripristinare e recuperare un VLDB se il contratto di servizio (SLA) è essenziale. A partire dalla versione 10g, Oracle ha introdotto una funzionalità RMAN che consente agli utenti di creare copie di immagini temporizzate dei file di dati del database Oracle su spazio di archiviazione su disco aggiuntivo situato sull'host del server DB. Queste copie di immagini possono essere aggiornate in modo incrementale tramite RMAN a intervalli definiti dall'utente. In caso di guasto, l'amministratore del database (DBA) può passare rapidamente dal supporto guasto alla copia dell'immagine del database Oracle, eliminando la necessità di un ripristino completo dei supporti del database. Il risultato è un SLA notevolmente migliorato, anche se al costo di raddoppiare lo storage del database richiesto.

Se sei interessato a un SLA per il tuo VLDB e stai pensando di spostare il database Oracle su un cloud pubblico come Google Cloud Platform (GCP), puoi configurare una struttura di protezione del database simile utilizzando risorse come Google Cloud NetApp Volumes (GCNV) per lo staging della copia dell'immagine del database in standby. GCNV è un servizio di archiviazione file ad alte prestazioni completamente gestito offerto da Google che semplifica, rende più sicuro ed economico migrare ed eseguire applicazioni aziendali complesse come il database Oracle in Google Cloud. I suoi diversi livelli di servizio consentono la flessibilità necessaria per controllare i costi di storage, ottenendo al contempo un RPO prossimo allo zero e un RTO gestibile. In questa documentazione, illustriamo come eseguire il provisioning ed esportare un file system NFS da un pool di archiviazione GCNV da montare su un server di database Oracle per lo staging di una copia del database in standby e il ripristino rapido in caso di guasto dello storage primario.

Questa soluzione risolve i seguenti casi di utilizzo:

* Backup della copia dell'immagine dei file di dati del database Oracle tramite RMAN su un file system NFS interno al server su un pool di archiviazione Google Cloud NetApp Volumes.
* Ripristino rapido di un Oracle VLDB in caso di guasto sulla stessa VM Google Compute Engine o su una VM di standby.
* Clona una copia del database Oracle su un host clone per altri casi d'uso dalla copia dell'immagine del database di staging in pochi minuti.




== Pubblico

Questa soluzione è destinata alle seguenti persone:

* Un DBA che configura e gestisce Oracle VLDB nella Google Cloud Platform.
* Architetto di soluzioni di database che testa i carichi di lavoro Oracle nella piattaforma Google Cloud.
* Un amministratore di storage che gestisce i database Oracle distribuiti nei pool di storage GCNV.
* Proprietario di un'applicazione che vorrebbe installare database Oracle in Google Cloud Platform.




== Ambiente di test e convalida della soluzione

I test e la convalida di questa soluzione sono stati eseguiti su pool di archiviazione Google Cloud NetApp Volumes e su VM Google Compute Engine che potrebbero non corrispondere all'ambiente di distribuzione finale. === Fattori chiave per l'implementazione

* *Livello di servizio del pool di storage GCNV.* Il pool di storage GCNV offre quattro livelli di servizio: Standard, Premium, Extreme e Flex. Per impostazione predefinita, a un volume creato all'interno di un pool di storage si applica un QoS automatico, che limita la velocità effettiva del volume in base alle sue dimensioni e al livello di servizio. L'unica eccezione è che i volumi allocati dal pool di storage Flex condividono la velocità effettiva disponibile per il pool di storage.
* *Limite di archiviazione per il backup in-server di Oracle VLDB per l'unione incrementale di RMAN.* Nei nostri test e validazioni, un volume NFS per il backup e l'unione incrementale di Oracle viene allocato da un singolo pool di storage GCNV con livello di servizio Standard, che ha un limite di capacità di 200 TiB. Per implementazioni oltre tale soglia, è possibile implementare più pool di storage GCNV in parallelo con più punti di montaggio NFS per offrire una maggiore capacità.
* *Recuperabilità di Oracle tramite merge incrementale RMAN.* Il backup e il merge incrementali RMAN in-server vengono generalmente eseguiti con una frequenza definita dall'utente in base agli obiettivi RTO e RPO. In caso di perdita totale dello storage primario dei file del database e/o dei log archiviati, si può verificare una perdita di dati. Il database Oracle può essere ripristinato fino all'ultimo backup incrementale disponibile dalla copia immagine di backup del database in-server. Per ridurre al minimo la perdita di dati, l'area di ripristino flash di Oracle deve essere impostata sul punto di montaggio del backup NFS GCNV e i log archiviati vengono sottoposti a backup sullo stesso mount NFS insieme alla copia immagine del database per eseguire il rollforward del database all'ultimo log archiviato in caso di errore. In questo modo, è possibile ottenere un RPO prossimo allo zero.
* *Esecuzione di Oracle VLDB tramite backup immagine NFS su pool di storage GCNV.* Al momento, la riassegnazione del livello di servizio del volume del database da standard a premium o extreme non è supportata. Tuttavia, la riassegnazione dei livelli di servizio da premium a extreme è supportata. Per eseguire il database su copia immagine per un periodo prolungato dopo un errore, si consiglia di allocare volume dal pool di storage premium per il backup della copia immagine del database.
* *VM di Google Compute Engine.* Nei test e nella convalida, abbiamo utilizzato istanze di VM GCE di tipo N1 come server di database Oracle. Esistono altri tipi di VM GCE che potrebbero essere più ottimizzati e adatti al carico di lavoro del database. Dimensionare la VM in modo appropriato in base al numero di vCPU e alla quantità di RAM, in base ai requisiti effettivi del carico di lavoro.
* *Configurazione dNFS.* dNFS è integrato nel kernel di Oracle ed è noto per aumentare significativamente le prestazioni del database Oracle quando Oracle viene distribuito su storage NFS. dNFS è incluso nel pacchetto binario di Oracle, ma non è attivato per impostazione predefinita. Deve essere attivato per qualsiasi implementazione di database Oracle su NFS. Per la distribuzione di più pool di storage GCNV per un VLDB, è necessario configurare correttamente i percorsi multipli di dNFS verso diversi pool di storage GCNV.




== Implementazione della soluzione

La soluzione presuppone che un Oracle VLDB sia già distribuito nell'ambiente Google Cloud all'interno di una VPC. L'Oracle VLDB può essere eseguito sullo storage GCNV o su qualsiasi altra opzione di storage all'interno dell'ecosistema Google Cloud. La sezione seguente fornisce procedure di deployment dettagliate per la configurazione di un backup dell'immagine Oracle VLDB e l'unione incrementale tramite RMAN su un file system NFS montato da uno storage pool GCNV. L'Oracle VLDB ha anche il suo storage primario ospitato su un diverso storage pool GCNV.

=== Montare un volume NFS da GCNV al server Oracle VLDB primario

[%collapsible%open]
====
Per ottenere prestazioni migliori, il volume GCNV predisposto per il backup in-server di Oracle VLDB dovrebbe idealmente trovarsi nella stessa zona Google Cloud del server DB primario e dello storage primario.

. Recupera il percorso di esportazione NFS del volume GCNV dalla console di Google Cloud.
+
image:gcp_ora_gcnv_inc_merge_vol_03.png["Questa immagine mostra la schermata dei volumi di archiviazione GCNV dalla console del portale Google Cloud"]

. Accedi al server Oracle VLDB primario. Come root, monta il volume NFS esportato dallo storage GCNV. Modifica l'indirizzo IP e il percorso del file del server NFS, se necessario.
+
[source, cli]
----
sudo mkdir /nfsgcnv
----
+
[source, cli]
----
mount 10.165.128.5:/orap-bkup /nfsgcnv -o rw,bg,hard,vers=3,proto=tcp,timeo=600,rsize=262144,wsize=262144
----
. Modificare la proprietà del punto di montaggio in oracle:oistall oppure modificare il nome utente Oracle e il gruppo primario, se necessario.
+
[source, cli]
----
sudo chown oracle:oinstall /nfsgcnv
----
. Aggiungere il montaggio NFS a /etc/fstab.
+
[source, cli]
----
sudo vi /etc/fstab
----

=== Imposta l'unione incrementale di Oracle RMAN per la copia dell'immagine su GCNV per il backup VLDB nel server

[%collapsible%open]
====

La copia dell'immagine del backup del database sarà aggiornata quanto la frequenza di esecuzione del backup/Unione incrementale. Pertanto, prendere in considerazione le performance del database, gli obiettivi RTO e RPO quando si decide la frequenza del backup incrementale e dell'Unione RMAN.

Lo storage primario è ospitato sul pool di storage GCNV con servizio premium. Anche in questo caso, questo non è un requisito e lo storage primario può essere ospitato su qualsiasi altra opzione di storage in Google Cloud.

. Creare un backup RMAN e uno script di Unione incrementale. Lo script alloca più canali per il backup e l'Unione di Parallel RMAN. La prima esecuzione genererebbe la copia iniziale completa dell'immagine di riferimento. In un'esecuzione completa, il reparto IT rimuove prima i backup obsoleti che si trovano al di fuori della finestra di conservazione per mantenere pulita l'area di staging. Il file di log corrente viene quindi commutato prima dell'Unione e del backup. Il backup incrementale segue l'Unione in modo che la copia dell'immagine del database sia in grado di eseguire il processo di recupero dello stato corrente del database con un ciclo di backup/Unione. L'ordine di Unione e backup può essere annullato per un ripristino più rapido in base alle preferenze dell'utente. Lo script RMAN può essere integrato in un semplice script della shell da eseguire da crontab sul server DB primario. Assicurarsi che l'autobackup del file di controllo sia attivo nell'impostazione RMAN.

Questo completa la configurazione di un backup di copia dell'immagine di standby Oracle VLDB e dell'unione su un montaggio NFS nel server dallo storage GCNV.

====


=== Ripristino rapido di Oracle VLDB tramite copia dell'immagine di backup



==== Recupero completo in caso di perdita o danneggiamento dei file di dati Oracle

[%collapsible%open]
====
In caso di guasto dovuto a un problema di archiviazione primaria, come perdita o danneggiamento dei dati, il database può essere rapidamente convertito in una copia dell'immagine sul montaggio NFS GCNV e ripristinato allo stato corrente senza doverlo ripristinare. L'eliminazione del ripristino dei supporti accelera enormemente il ripristino del database per un VLDB. Questo caso di utilizzo presuppone che il server Oracle VLDB DB sia intatto e che il file di controllo del database, i registri archiviati e quelli correnti siano tutti disponibili per il ripristino.

=== Clonare un Oracle VLDB per altri casi d'uso dalla copia dell'immagine di backup

[%collapsible%open]
====
Clonare un nuovo Oracle VLDB dalla copia dell'immagine di backup su GCNV è simile al ripristino point-in-time del database primario su un nuovo server DB. Il team di NetApp Solutions Engineering fornisce un toolkit di automazione per semplificare l'intero processo, garantendo un'implementazione rapida (in pochi minuti) e senza sforzi.

Il seguente toolkit di automazione contiene un playbook che può essere utilizzato per creare o aggiornare cloni di un Oracle VLDB da una copia dell'immagine di backup del database primario archiviata nello storage GCNV tramite montaggio NFS per una gestione completa del ciclo di vita del clone. Per i dettagli fare riferimento alla documentazione: link:https://docs.netapp.com/us-en/netapp-solutions/databases/automation_ora_bkup_clone.html["Ciclo di vita automatizzato di clonazione Oracle VLDB con backup dell'immagine nel server"^] .

----
https://bitbucket.ngage.netapp.com/mvc/projects/NS-BB/repos/na_oracle_bkup_clone/browse
----

NOTE: Al momento è possibile accedere al toolkit solo da parte dell'utente interno NetApp con accesso bitbucket. Per gli utenti esterni interessati, richiedere l'accesso al proprio account team o contattare il team di progettazione delle soluzioni NetApp.
